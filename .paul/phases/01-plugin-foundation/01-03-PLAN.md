<objective>
## Goal
Create a database access layer (`db.py`) that wraps the GRAMPS database with helpers used by all four tools, and a background worker pattern (`worker.py`) that runs scans in a thread so the GTK main loop never freezes — even on databases with tens of thousands of people.

## Purpose
Every tool in Phases 2–5 needs to iterate people and families, resolve names, and push results into a ResultList without blocking the UI. Building this infrastructure once in Phase 1 means each tool phase only writes the scan logic — not the threading boilerplate.

## Output
- `grampsclean/db.py` — `GrampsDb` wrapper: person/family iteration, name formatting, event date extraction
- `grampsclean/worker.py` — `ScanWorker`: runs a scan function in a thread, reports rows via GLib idle callbacks, updates status on completion
- `grampsclean/tool.py` — updated: stores `self.db_wrap` (GrampsDb instance) for use by tool tabs
</objective>

    **get_event_date(self, person, event_type: str):**

    - Get the ref: `person.get_birth_ref()` or `person.get_death_ref()`

    - Get event: `self.db.get_event_from_handle(ref.ref)`
    - If event is None: return None
    - Get date: `event.get_date_object()`
    - If date is None or `not date.is_regular()`: return None
    - Return the Date object

    **iter_people(self):**
    - Yield each Person: `yield from self.db.iter_people()`

    **iter_families(self):**
    - Yield each Family: `yield from self.db.iter_families()`

    **get_family_from_handle(self, handle):**
    - Return `self.db.get_family_from_handle(handle)`

    **get_person_from_handle(self, handle):**
    - Return `self.db.get_person_from_handle(handle)`

<task type="auto">
  <name>Task 2: Create ScanWorker background thread pattern</name>
  <files>grampsclean/worker.py</files>
  <action>
    Create grampsclean/worker.py with one class: `ScanWorker`.

    Imports needed:
    - `import threading`
    - `from gi.repository import GLib`

    **ScanWorker:**
    - __init__(self, result_list, scan_fn, db_wrap):
      - `self.result_list` — the ResultList widget to populate
      - `self.scan_fn` — a callable that takes (db_wrap) and yields
        (row_tuple, status_text) pairs
        - row_tuple: tuple of strings matching ResultList columns
        - status_text: string shown in status bar after scan completes
      - `self.db_wrap` — GrampsDb instance
      - `self._thread = None`
      - `self._cancelled = False`

    **start(self):**
    - If a thread is already running: return immediately
    - Call `self.result_list.clear()`
    - Call `self.result_list.set_status("Scanning…")`
    - Create and start a daemon thread running `self._run`
    - Store as `self._thread`

    **cancel(self):**
    - Set `self._cancelled = True`

    **_run(self):**
    - This runs in the background thread
    - Call `self.scan_fn(self.db_wrap)` — it returns a generator
    - For each `(row_tuple, status_text)` yielded:
      - If `self._cancelled`: break
      - Schedule row append on main thread:
        `GLib.idle_add(self.result_list.append_row, *row_tuple)`
    - After loop completes (or cancelled):
      - If cancelled: `GLib.idle_add(self.result_list.set_status, "Scan cancelled")`
      - Else: `GLib.idle_add(self.result_list.set_status, status_text)`
    - Wrap the entire body in try/except Exception:
      - On exception: `GLib.idle_add(self.result_list.set_status, f"Error: {e}")`

    Note: scan_fn generators should yield (row_tuple, final_status) where
    status_text on the LAST yield becomes the completion message.
    For intermediate yields, status_text can be "" or a progress string.

    No GRAMPS imports in this file — only threading, GLib, and the
    result_list/scan_fn/db_wrap passed in.
  </action>
  <verify>python3 -m py_compile grampsclean/worker.py && echo OK</verify>
  <done>AC-3 and AC-4 structurally satisfied; AC-5 partially satisfied</done>
</task>
