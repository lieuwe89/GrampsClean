<objective>
## Goal
Replace the flat ResultList in ImpossibilitiesTab with a grouped TreeView that organises findings under expandable rule-name headers, colour-codes Error/Warning rows, and surfaces the per-finding detail string.

## Purpose
The ROADMAP deliverable for Phase 4 is "results grouped by rule type, navigate to person/family". The flat list from 04-01 works but is hard to read when many rules fire across a large database. Grouping by rule lets users quickly see which categories have the most findings and expand/collapse sections.

## Output
- New `GroupedResultView` class in `tab_impossibilities.py` — Gtk.TreeStore-based, same clear/set_status/append_row API as ResultList so ScanWorker drives it unchanged
- ImpossibilitiesTab uses GroupedResultView instead of ResultList
- _scan_fn yields 6-tuples (adds detail field); rule groups auto-expand as findings arrive; Error rows red, Warning rows amber
- COLUMNS constant removed (GroupedResultView owns its column config)
</objective>

<task type="auto">
  <name>Task 1: Add GroupedResultView class</name>
  <files>grampsclean/tab_impossibilities.py</files>
  <action>
    Add the `GroupedResultView` class immediately above the `ImpossibilitiesTab` class (after the rule engine functions, before the tab class separator comment).

    **TreeStore columns (internal):**
    | Index | Name | Type | Purpose |
    |-------|------|------|---------|
    | 0 | display | str | rule_name for parent; person_name for child |
    | 1 | gramps_id | str | empty for parent; person GRAMPS ID for child |
    | 2 | severity | str | empty for parent; "Error"/"Warning" for child |
    | 3 | detail | str | empty for parent; finding detail string for child |
    | 4 | handle | str | empty for parent; person_handle for child (hidden) |
    | 5 | color | str | "" for parent (default color); "#cc0000" for Error; "#c07000" for Warning |
    | 6 | weight | int | 700 (bold) for parent; 400 (normal) for child |

    **GTK columns (visible):**
    - "Rule / Name" — col 0 text, col 5 foreground, col 6 weight; min_width=200, resizable
    - "ID" — col 1 text; min_width=80, resizable
    - "Severity" — col 2 text, col 5 foreground; min_width=80, resizable
    - "Detail" — col 3 text; min_width=200, resizable

    **Public methods (same API surface as ResultList):**

    `clear(self)`:
    - `self.store.clear()`
    - `self._rule_iters = {}` — clear rule → TreeIter map
    - `self._rule_counts = {}` — clear rule → count map
    - Reset status_label to `_STATUS_DEFAULT`

    `set_status(self, text)`:
    - `self.status_label.set_text(text)`

    `append_row(self, name, gramps_id, rule_name, severity, detail="", handle="")`:
    - If `not rule_name`: return immediately (empty sentinel from no-findings yield)
    - If `rule_name` not in `self._rule_iters`:
      - Create parent row: `self.store.append(None, [rule_name, "", "", "", "", "", 700])`
      - Store iter in `self._rule_iters[rule_name]`; set `self._rule_counts[rule_name] = 0`
    - Increment `self._rule_counts[rule_name]`
    - Update parent label: `self.store.set_value(parent_iter, 0, f"{rule_name} ({count})")`
    - `color = "#cc0000" if severity == "Error" else "#c07000"`
    - Append child: `self.store.append(parent_iter, [name, gramps_id, severity, detail, handle, color, 400])`
    - Expand parent: `self.tree.expand_row(self.store.get_path(parent_iter), False)`
      - Note: `append_row` is called via `GLib.idle_add` (GTK main thread) — safe to call GTK methods here

    **Private method:**

    `_row_activated(self, tree, path, column)`:
    - `it = model.get_iter(path)`
    - `handle = model.get_value(it, 4)` — col 4 is handle
    - If handle and `self._on_activate`: call it

    **Layout:** Gtk.Box vertical, spacing=4:
    - Scrolled window (auto/auto policy, ShadowType.IN) containing the TreeView — expand=True
    - Status label (halign=START, margin_start=4, margin_bottom=4) — expand=False

    Avoid: calling GTK methods from background threads. `append_row` is called via `GLib.idle_add` and is safe. Do not add any threading logic here.
