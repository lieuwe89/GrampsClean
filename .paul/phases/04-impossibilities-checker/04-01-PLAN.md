<acceptance_criteria>

## AC-1: Snapshot captures all data needed for impossibility checks
```gherkin
Given: A GRAMPS database is loaded
When: _build_snapshot() runs on the main thread
Then: Returns a dict with:
  - "persons": {handle: {name, gramps_id, birth, death, burial, birth_count,
                         death_count, life_event_dates, parent_family_handles,
                         child_family_handles}}
  - "families": {handle: {gramps_id, father_handle, mother_handle, child_handles,
                           marriage, divorce}}
  where dates are (year, month, day) tuples or None, counts are ints,
  handles are lists of strings
```

## AC-2: All 15 rules detect their target violations
```gherkin
Given: A snapshot containing persons with various date and relationship issues
When: run_rules(snapshot, thresholds) is called
Then: Returns a list of finding dicts [{person_handle, family_handle, rule_id,
       rule_name, severity, detail}] where:
  - Each of the 15 rules fires for its specific violation
  - Severity is "Error" for definite impossibilities, "Warning" for implausible-but-possible
  - Rules with missing dates on either side produce no finding (can't confirm violation)
```

## AC-3: ImpossibilitiesTab scans and displays findings
```gherkin
Given: GrampsClean tool is open with the Impossibilities tab active
When: User clicks Scan
Then: Background scan runs without freezing the UI, findings appear in the
      ResultList (columns: Name, ID, Rule, Severity), and the status bar
      shows a count summary like "23 issues found (8 errors, 15 warnings)"
```

    **Date extraction helper — add `_date_tuple(date_obj)` inside the module:**
    - Accepts a GRAMPS Date object (or None)
    - Returns (year, month, day) tuple if date is regular and has a year, else None
    - month and day default to 0 if not set (year-only dates)
    - Do NOT call this from a worker thread — only used during snapshot building

    - name: db_wrap.format_name(person)
    - gramps_id: db_wrap.get_gramps_id(person)

    - death: _date_tuple of death event date
    - burial: extract burial event separately — iterate person.get_event_ref_list(),
      for each ref get the event, check event.get_type() == EventType.BURIAL,
      use the first one found; apply _date_tuple
    - birth_count: count of events where type == EventType.BIRTH
    - death_count: count of events where type == EventType.DEATH
    - life_event_dates: list of _date_tuple for ALL events that are NOT birth/death/burial
      and have a valid date tuple (skip Nones)
    - parent_family_handles: list(person.get_family_handle_list()) — families as parent
    - child_family_handles: list(person.get_parent_family_handle_list()) — families as child

    **Families sub-dict — for every family in db_wrap.iter_families():**
    - gramps_id: family.get_gramps_id()
    - father_handle: family.get_father_handle() or None
    - mother_handle: family.get_mother_handle() or None
    - child_handles: [ref.ref for ref in family.get_child_ref_list()]
    - marriage: _date_tuple of marriage event — iterate family.get_event_ref_list(),
      find type == EventType.MARRIAGE, use first found
    - divorce: _date_tuple of divorce event — same but EventType.DIVORCE

    Imports needed at top: `from gramps.gen.lib import EventType`
    Avoid: storing GRAMPS Person/Family/Event objects in the snapshot.
  </action>

<task type="auto">
  <name>Task 2: Implement all 15 impossibility rules in run_rules()</name>
  <files>grampsclean/tab_impossibilities.py</files>
  <action>
    Add `run_rules(snapshot, thresholds)` to `tab_impossibilities.py`.
    Returns a list of finding dicts (plain Python, safe for use in worker thread).

    Each finding: {"person_handle": str, "family_handle": str or None,
                   "rule_id": str, "rule_name": str, "severity": str, "detail": str}

    `thresholds` is a dict with defaults (hardcoded for now, Phase 04-02 adds UI):
    ```python
    DEFAULT_THRESHOLDS = {
        "father_postdeath_years": 1,   # child born >1yr after father death = Error
        "mother_min_age": 12,
        "father_min_age": 13,
        "mother_max_age": 60,
        "father_max_age": 90,
    }
    ```

    **Date comparison helper — `_cmp(a, b)`:**
    - Compares two (year, month, day) tuples; returns -1/0/1 like cmp
    - If month or day is 0, compare year only (year-level precision)
    - If both have non-zero month: compare (year, month); if both have non-zero day: (year, month, day)

    **Rule implementation — iterate persons and families:**

    Person-level rules (iterate snapshot["persons"].items()):
    1. **R01 death-before-birth** (Error): death < birth → flag person
    2. **R02 burial-before-death** (Error): burial < death → flag person
    3. **R03 duplicate-birth** (Error): birth_count > 1 → flag person
    4. **R04 duplicate-death** (Error): death_count > 1 → flag person
    5. **R05 event-before-birth** (Warning): any life_event_date < birth → flag person,
       detail shows count of such events
    6. **R06 event-after-death** (Warning): any life_event_date > death → flag person,
       detail shows count of such events

    Family-level rules (iterate snapshot["families"].items()):
    For each family, look up father/mother/children in snapshot["persons"]:
    7. **R07 child-before-parent-born** (Error): for each child, if child.birth < father.birth
       OR child.birth < mother.birth → flag child person, detail names the parent
    8. **R08 child-after-father-death** (Error): child.birth year > father.death year +
       thresholds["father_postdeath_years"] → flag child person
    9. **R09 child-after-mother-death** (Error): child.birth year > mother.death year → flag child
    10. **R10 parent-too-young** (Warning): (child.birth year - parent.birth year) <
        min_age threshold → flag child person, detail names parent and age difference
    11. **R11 parent-too-old** (Warning): (child.birth year - parent.birth year) >
        max_age threshold → flag child person, detail names parent and age difference
    12. **R12 marriage-before-spouse-birth** (Error): family.marriage < spouse.birth for
        either spouse → flag family (use father_handle as person_handle, family_handle set)
    13. **R13 marriage-after-spouse-death** (Error): family.marriage > spouse.death for
        either spouse → flag family
    14. **R14 overlapping-marriages** (Warning): a person appears as parent (father/mother)
        in 2+ families, both families have a marriage date, and neither has a divorce date →
        flag the person (person_handle), detail lists family gramps_ids; skip if one marriage
        post-dates the other's divorce
    15. **R15 circular-ancestry** (Error): detect if a person appears as both an ancestor
        and a descendant in the family graph — use iterative DFS on child_family_handles →
        child_handles chain; if any person_handle is visited twice in a root-to-leaf path,
        flag it; limit depth to 50 to avoid infinite loops on genuinely circular data

    **For all rules:** Skip if required dates are None (can't confirm violation without data).
    Do NOT call any GRAMPS db methods — snapshot only.
    Return the full findings list sorted by rule_id, then person name.
  </action>
  <verify>With a real database, run the scan and confirm each rule fires appropriately.
    A database with no data errors should return 0 findings.
    Introduce a test person with death_year &lt; birth_year and confirm R01 fires.</verify>
